# Docker 학습 가이드 (권장 순서)

이 문서는 **입문 → 실습 → Dockerfile → Docker Compose** 순으로 읽고 따라 하면 됩니다.  
프로젝트에서 DB만 Compose로 띄우는 방법은 [`DOCKER_README.md`](./DOCKER_README.md)를 참고하세요.

## 0. 한 줄 정리

| 도구 | 역할 |
|------|------|
| **Docker** | 앱과 의존성을 **이미지**로 묶고 **컨테이너**로 실행해 환경을 통일한다. |
| **Docker Compose** | 여러 컨테이너를 **YAML 한 파일**로 정의하고 한 번에 실행한다. |

## Docker 용어·개념 쉬운 설명 (비개발자용)

**비유로 잡기**: **이미지**는 “설계도 + 부품이 들어 있는 상자(복사해 쓰는 원본)”에 가깝고, **컨테이너**는 그 상자를 한 번 펼쳐서 **실제로 돌아가게 설치한 방 하나**에 가깝습니다. 같은 설계도로 방을 여러 개 만들 수 있습니다.

아래는 Docker 문서·대화에서 자주 나오는 말을 **짧게만** 정리한 것입니다.

### 전체를 돌아주는 것

| 용어 | 쉬운 설명 |
|------|-----------|
| **Docker** | 프로그램을 “정해진 박스(컨테이너)” 안에서 돌리게 해 주는 도구 묶음. 내 PC나 서버가 달라도 비슷하게 동작하게 맞추는 데 쓴다. |
| **Docker Desktop** | Mac/Windows에서 Docker를 쓰기 쉽게 만든 **앱**. 설치하면 터미널에서 `docker` 명령을 쓸 수 있다. |
| **Docker Engine** | 실제로 컨테이너를 만들고 돌리는 **배후 프로그램**(서버에만 깔리는 경우도 많다). |
| **데몬(daemon)** | 사용자가 안 보는 곳에서 계속 떠 있는 **백그라운드 서비스**. Docker Engine이 이런 식으로 동작한다고 보면 된다. |

### 꾸러미와 실행

| 용어 | 쉬운 설명 |
|------|-----------|
| **이미지(Image)** | 앱 실행에 필요한 파일·설정이 담긴 **읽기 전용 꾸러미**. 설치 CD나 “복사용 원본”에 가깝다. |
| **컨테이너(Container)** | 이미지를 **한 번 실행한 상태**. 꺼졌다 켜졌다 할 수 있는 **살아 있는 인스턴스**다. |
| **태그(Tag)** | 이미지 이름 뒤의 버전 표시. 예: `postgres:16`에서 `16`은 “16번 버전”이다. `latest`는 “지금 기본으로 가리키는 최신”이라는 뜻에 가깝다. |
| **Docker Hub** | 이미지를 **올려두고 내려받는** 대표적인 저장소(웹사이트). 공개 이미지를 많이 찾는다. |
| **레지스트리(Registry)** | 이미지를 보관·배포하는 **서버**. Hub 말고 회사 전용 등 다른 곳도 있다. |
| **docker pull** | 레지스트리에서 이미지를 **내 PC로 받아오기**. |
| **docker push** | 내가 만든 이미지를 레지스트리에 **올리기**. |
| **docker run** | 이미지로 **컨테이너를 만들고 실행**하기. 없는 이미지면 받은 뒤 실행한다. |
| **docker ps** | **지금 돌아가는** 컨테이너 목록 보기. |

### 직접 만들기

| 용어 | 쉬운 설명 |
|------|-----------|
| **Dockerfile** | 이미지를 **어떤 순서로 만들지** 적어 둔 텍스트 파일. 요리 레시피에 가깝다. |
| **docker build** | Dockerfile을 읽어서 **이미지를 빌드(생성)**한다. |
| **빌드(Build)** | 레시피대로 재료를 모아 **설치 가능한 꾸러미(이미지)**를 만드는 과정. |
| **베이스 이미지(Base image)** | Dockerfile 맨 위 `FROM`에서 시작하는 **바닥부터 쓰는 이미지**. 예: `node:20-alpine`. |
| **레이어(Layer)** | 이미지를 **겹겹이 쌓아** 만든 구조. 바꾼 부분만 다시 받거나 캐시해서 속도를 낸다. |
| **멀티 스테이지 빌드** | **만드는 단계**와 **실행만 하는 단계**를 나눠서, 최종 이미지를 가볍게 만드는 방식. |
| **.dockerignore** | 이미지에 **넣지 않을 파일** 목록(`.gitignore`와 비슷한 역할). |

### Dockerfile 안에서 나오는 말

| 용어 | 쉬운 설명 |
|------|-----------|
| **FROM** | **어떤 이미지에서 시작**할지 정한다. |
| **WORKDIR** | 컨테이너 안에서 **작업 폴더**를 정한다. |
| **COPY** | 내 PC(빌드할 때 넘기는 폴더)에서 파일을 **이미지 안으로 복사**한다. |
| **RUN** | 이미지를 만들 **도중에** 한 번 실행하는 명령(예: 패키지 설치). |
| **CMD** | 컨테이너가 **시작될 때 기본으로 실행**할 명령. |
| **ENTRYPOINT** | 컨테이너의 **진입점**—보통 항상 실행되는 실행 파일을 고정할 때 쓴다. `CMD`와 함께 쓰는 경우가 많다. |

### PC와 연결하기

| 용어 | 쉬운 설명 |
|------|-----------|
| **호스트(Host)** | Docker를 돌리는 **실제 컴퓨터**(내 맥북, 회사 서버 등). 컨테이너는 그 위에서 손님 방처럼 떠 있다. |
| **포트(Port)** | 네트워크로 **어느 문(채널)으로 들어올지** 정하는 **숫자**. 같은 PC에서도 프로그램마다 다른 포트를 쓰면 서로 섞이지 않는다. 웹은 보통 80/443, PostgreSQL은 보통 **5432**를 쓴다. |
| **포트 매핑** | **호스트 쪽 문 번호**와 **컨테이너 안쪽 문 번호**를 짝지어 주는 것. 형식은 항상 `호스트:컨테이너`(왼쪽이 내 PC, 오른쪽이 컨테이너). 예: `5432:5432`는 “내 PC의 5432번으로 오는 연결을, 그 컨테이너 안의 5432번 서비스로 넘긴다”는 뜻이다. |

### 데이터·설정 유지

| 용어 | 쉬운 설명 |
|------|-----------|
| **볼륨(Volume)** | 컨테이너 밖에 두는 **저장 공간**에 가깝다. 컨테이너를 지워도 이름 있는 볼륨에 넣어 둔 데이터는 남을 수 있다. |
| **마운트(Mount)** | 호스트의 폴더나 볼륨을 컨테이너 안 특정 경로에 **붙이는 것**. |
| **바인드 마운트(Bind mount)** | 내 PC의 **특정 폴더**를 컨테이너에 직접 연결하는 방식. 개발할 때 코드를 바로 반영하려고 쓰기도 한다. |

### 여러 컨테이너·Compose

| 용어 | 쉬운 설명 |
|------|-----------|
| **Docker Compose** | **여러 개**를 한 번에 정의하고 `docker compose up` 한 방에 띄우는 도구. |
| **compose 파일** | 보통 `docker-compose.yml` 또는 `compose.yaml`. **설정을 글로 적어 둔 파일**이다. |
| **YAML** | 들여쓰기로 구조를 나누는 **설정 파일 형식**. 사람이 읽기 쉽게 생겼다. |
| **서비스(Service)** | Compose에서 말하는 **역할 하나**(예: `postgres` DB 서비스). 보통 서비스 하나가 컨테이너 하나에 대응한다. |
| **docker compose up** | compose 파일대로 **서비스들을 실행**한다. |
| **docker compose down** | compose로 띄운 것들을 **내리고 정리**한다. |
| **environment / 환경 변수** | 컨테이너 안 프로그램에 넘기는 **설정값**(비밀번호, DB 이름 등). compose의 `environment:`에 적을 수 있다. |
| **container_name** | 컨테이너에 붙이는 **고정 이름**. 목록에서 찾기 쉽다. |
| **restart 정책** | 예: `always`는 컴퓨터를 재부팅해도 **다시 자동으로** 띄우겠다는 뜻에 가깝다. |
| **네트워크(Network)** | 컨테이너끼리 **이름으로 서로 부를 수 있게** 묶어 주는 가상 네트워크. Compose는 보통 프로젝트 단위로 하나 만든다. |

### 그 밖에 자주 보는 말

| 용어 | 쉬운 설명 |
|------|-----------|
| **이미지 / 컨테이너 정리** | 안 쓰는 이미지·컨테이너를 지워 **디스크를 비우는** 작업. `docker system prune` 등으로 할 수 있다(주의: 필요한 것까지 지울 수 있음). |
| **Alpine** | 용량이 작은 Linux를 넣은 이미지 계열. `node:20-alpine`처럼 **가볍게** 쓰려고 붙는 이름이다. |

위 표를 한 번 훑은 뒤 **1단계**부터 실습으로 넘어가면 전체 흐름이 잘 잡힌다.

## 1단계: 개념 익히기

다음 네 가지를 먼저 구분하면 이후 실습이 수월합니다.

### 1.1 이미지(Image) vs 컨테이너(Container)

- **이미지**: 실행에 필요한 파일·설정이 담긴 **읽기 전용 템플릿** (예: `postgres:16`, `node:20-alpine`).
- **컨테이너**: 이미지를 **실행한 인스턴스** — 프로세스가 돌아가는 격리된 환경.

**“이미지에서 컨테이너를 여러 개?”가 헷갈릴 때**  
맞는 이해는 이렇게 잡으면 된다. **이미지는 바뀌지 않는 원본(설계도)**이고, **`docker run`을 할 때마다 그 원본을 복사해 올린 뒤 돌아가게 만든 것이 컨테이너 하나**다. 그래서 **같은 이미지(같은 원본)를 기준으로** `docker run`을 여러 번 하면 **서로 다른 컨테이너가 여러 개** 생긴다(각각 ID가 다르고 메모리·디스크도 분리된다). “이미지가 컨테이너를 여러 개 낳는다”가 아니라, **“같은 설계도로 방을 여러 칸 짓는 것”**에 가깝다.

### 1.2 포트 매핑(Port mapping)

**왜 필요한가**  
컨테이너는 **겹치지 않는 작은 네트워크 공간** 안에서만 듣고 있다고 보면 된다. 그래서 “내 맥/윈도우에서 브라우저나 DB 클라이언트로 접속”하려면, **호스트(실제 PC)의 어떤 번호로 들어온 요청을 컨테이너 안의 번호로 넘겨 주는 연결**이 필요하다. 그 연결이 **포트 매핑**이다. 매핑을 안 하면 컨테이너 안에서만 서로 통신할 수 있고, 밖에서 `localhost`로는 보통 닿지 않는다.

**형식: 항상 `호스트포트:컨테이너포트`**

| 부분 | 의미 |
|------|------|
| **앞(왼쪽)** | **내 PC(호스트)**에서 열어 둘 번호. 여기로 접속한다. |
| **뒤(오른쪽)** | **컨테이너 안**에서 이미 그 서비스가 듣고 있는 번호(이미지·앱이 정한 값). |

예: PostgreSQL 공식 이미지는 컨테이너 **안에서** 보통 **5432**로 받는다. 그래서 `5432:5432`는 “내 PC의 5432번으로 들어오면 → 그 컨테이너의 PostgreSQL(5432)로 전달”이다. 앱 설정에서 `DB_HOST=localhost`, `DB_PORT=5432`라고 하면 **호스트 쪽 5432**를 가리키게 된다.

**숫자를 다르게 쓰는 예 (`5433:5432`)**  
내 PC에 이미 다른 PostgreSQL이 **5432**를 쓰고 있으면, 호스트 쪽 **5432**는 비울 수 없다. 그때는 예를 들어 **`5433:5432`**처럼 적는다. 의미는 “내 PC **5433**으로 들어오면 → 컨테이너 **안**의 **5432**(Postgres)로 넘긴다”다. 앱에서는 `DB_PORT=5433`으로 맞춘다.

**정리**

- 포트 매핑은 **밖(호스트) ↔ 안(컨테이너)**을 이어 주는 **다리**이고, **왼쪽이 내 PC, 오른쪽이 컨테이너**라고 기억하면 헷갈리지 않는다.
- `docker run -p 8080:80` / Compose의 `ports: - '5432:5432'` 모두 같은 규칙이다.
- 호스트에서 쓰는 **왼쪽 번호**가 이미 다른 프로그램과 겹치면 **포트 충돌**이 나므로, 왼쪽만 바꿔 주면 된다.

### 1.3 볼륨(Volume)

컨테이너를 지워도 **데이터를 유지**하려면 디스크를 호스트나 이름 있는 볼륨에 마운트합니다.  
이 프로젝트의 `postgres_data`는 PostgreSQL 데이터 디렉터리를 그렇게 보존합니다.

### 1.4 Dockerfile (미리 알아두기)

나중에 3단계에서 다룹니다. **이미지를 어떻게 빌드할지** 적는 텍스트 파일입니다.

**체크리스트**

- [ ] 이미지와 컨테이너의 차이를 말할 수 있다.
- [ ] 포트 매핑이 왜 필요한지 안다.
- [ ] DB 데이터를 컨테이너만 쓰면 왜 사라질 수 있는지, 볼륨이 왜 필요한지 안다.

## 2단계: 실습 (공식 흐름 따라가기)

Docker Desktop 설치 후 터미널에서 진행합니다.

### 2.1 설치 확인

```bash
docker --version
docker compose version
```

### 2.2 가장 작은 실행: hello-world

```bash
docker run hello-world
```

이미지가 없으면 받아온 뒤 실행되며, Docker 파이프라인이 정상인지 확인할 수 있습니다.

### 2.3 간단한 웹 서버 (선택)

```bash
docker run -d -p 8080:80 --name nginx-demo nginx:alpine
```

브라우저에서 `http://localhost:8080` 접속 후 컨테이너 정리:

```bash
docker stop nginx-demo
docker rm nginx-demo
```

### 2.4 공식 가이드로 확장

- [Docker — Guides](https://docs.docker.com/guides/) 에서 **Get started** 계열 튜토리얼을 순서대로 진행하는 것을 권장합니다.

**체크리스트**

- [ ] `docker run`으로 이미지를 받아 실행해 본다.
- [ ] `-p`로 포트를 열어 호스트에서 서비스에 접속해 본다.
- [ ] `docker ps`, `docker stop`, `docker rm`의 역할을 안다.

## 3단계: Dockerfile 익히기 (Node / NestJS 관점)

커스텀 이미지를 만들 때는 프로젝트 루트에 `Dockerfile`을 두고 `docker build`로 빌드합니다.

### 3.1 자주 쓰는 지시문 순서

| 지시문 | 역할 |
|--------|------|
| `FROM` | 베이스 이미지 (예: `node:20-alpine`) |
| `WORKDIR` | 이후 명령의 작업 디렉터리 |
| `COPY` | 빌드 컨텍스트에서 파일·폴더 복사 |
| `RUN` | 이미지 빌드 시 실행 (의존성 설치 등) |
| `CMD` / `ENTRYPOINT` | 컨테이너 **시작 시** 실행할 기본 명령 |

Nest는 보통 `npm ci` 또는 `npm install` → `npm run build` → 프로덕션에서는 `node dist/main.js` 형태로 실행합니다. 멀티 스테이지 빌드로 빌드용 이미지와 실행용 이미지를 나누는 패턴도 많이 씁니다.

### 3.2 참고 자료

- [Dockerfile reference](https://docs.docker.com/reference/dockerfile/)
- [Node.js Docker 이미지](https://hub.docker.com/_/node)

**체크리스트**

- [ ] `FROM` → `WORKDIR` → `COPY` → `RUN` → `CMD` 흐름을 설명할 수 있다.
- [ ] `RUN`(빌드 시)과 `CMD`(실행 시)의 차이를 안다.

## 4단계: Docker Compose

### 4.1 Compose가 하는 일

- 여러 **서비스**(컨테이너)를 한 파일에 선언한다.
- `docker compose up`으로 한꺼번에 띄리고, 네트워크·볼륨을 묶어 관리하기 쉽다.

### 4.2 이 프로젝트에서의 위치

현재 Rabbit Tracker BE는 **PostgreSQL만** Compose로 실행하는 구성입니다. Nest 앱은 호스트에서 `npm run start:dev` 등으로 돌리는 흐름과 잘 맞습니다.

- Compose 파일: 루트의 [`docker-compose.yml`](./docker-compose.yml)
- 실행·`.env` 연동·트러블슈팅: [`DOCKER_README.md`](./DOCKER_README.md)

앱까지 컨테이너로 옮기려면 Dockerfile을 추가하고 `docker-compose.yml`에 `app` 서비스를 정의하면 됩니다. 팀 규모·배포 방식에 따라 선택하면 됩니다.

### 4.3 유용한 명령

```bash
docker compose up -d          # 백그라운드 실행
docker compose ps
docker compose logs -f postgres
docker compose down           # 컨테이너 중지·삭제 (볼륨은 기본 유지)
```

**체크리스트**

- [ ] `services`, `ports`, `environment`, `volumes`가 compose 파일에서 무엇을 의미하는지 안다.
- [ ] 이 레포에서 DB를 띄우고 Nest와 연결하는 절차를 [`DOCKER_README.md`](./DOCKER_README.md)로 수행해 본다.
